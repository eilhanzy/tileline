# Tileline Pre-Beta Plan (FSR 1.0 + Input)

Date: 2026-03-14

## Goal

Ship a pre-beta baseline that keeps TLApp-class scenes stable while improving accessibility:

- consistent 60 FPS target policy
- scalable image quality via FSR 1.0
- full keyboard combo layer
- gamepad parity for camera/gameplay triggers

## Why FSR 1.0 First

FSR 1.0 is the fastest low-risk uplift path for pre-beta:

- spatial upscaler + sharpening (EASU + RCAS)
- broad GPU compatibility
- easy insertion into existing post-process chain
- low integration complexity compared to temporal upscalers

Trade-off:

- thin geometry shimmering and temporal instability can remain in motion.

## Milestones

1. Render Scale Backbone
- add runtime `render_scale` (0.5..1.0)
- quality presets: `UltraQuality`, `Quality`, `Balanced`, `Performance`
- expose CLI + script hooks for profile switching

2. FSR 1.0 Integration
- add EASU + RCAS fullscreen passes in runtime renderer path
- keep fallback native render path when disabled
- add per-platform default preset selection

3. Adaptive Quality Controller
- EMA frame-time based scale control
- avoid oscillation with hysteresis and min dwell time
- target 60 FPS under vsync-on/off modes

4. Input Accessibility (now started)
- keyboard combo layer (`Ctrl+Q/F/R/L`, `Alt+Enter`)
- expanded movement map (WASD + arrows + E/Q)
- gamepad movement/look/action parity

5. Validation Matrix
- RTX dGPU (Vulkan)
- Ryzen iGPU (Vulkan)
- Apple Silicon (Metal/UMA)
- Panthor/RK3588 class (Vulkan)

6. Packaging and Distribution
- add `.pak` asset packaging flow for demo/runtime content
- provide `pack/list/unpack` tooling for CI + release scripts
- validate deterministic pack output and checksum verification on unpack

## Current Gaps Before Pre-Beta Freeze

- FSR 1.0 pass is not yet integrated into `runtime/src/wgpu_scene_renderer.rs`.
- Input action mapping is demo-local (`tlapp_app`) and should move to reusable runtime input module.
- Gamepad haptics and remapping UI are not present.
- Graphics quality presets are not yet surfaced in `.tlscript` / `.tlsprite` APIs.
- Networked input prediction/reconciliation is pending for NPS-driven multiplayer scenarios.
- `.pak` loading is not yet wired into TLApp runtime asset resolver (tooling exists, runtime mounting pending).

## Exit Criteria (Pre-Beta)

- TLApp demo sustains target FPS profile with deterministic physics behavior.
- Keyboard + gamepad controls are available by default and documented.
- FSR 1.0 can be toggled live and yields measurable FPS uplift on at least 3 hardware classes.
- Reproducible release artifacts are generated by script and documented.
