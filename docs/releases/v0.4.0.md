# Tileline v0.4.0 Release Notes

Release date: 2026-03-18

## Scope

`v0.4.0` is a performance release focused on two complementary goals:

1. **Parallelise every remaining serial hot path** in the ParadoxPE physics
   engine (broadphase, narrowphase, solver, sleep).
2. **Overlap CPU physics with GPU upload** using an MPS async pipeline so
   the main thread is never blocked waiting for both.

All baselines below are measured at **10 000 dynamic bodies / 2 substeps**
on a Ryzen 9 7900 (24 threads) running the `tlapp` bounce showcase.

---

## ParadoxPE Engine — Parallelisation Sweep

### Broadphase: parallel speculative AABB sweep

The speculative-sweep pass that expands each dynamic body's AABB by its
predicted velocity was a serial loop over all bodies. It is now driven by
`par_iter_mut` so all cores participate.

| Metric | Before | After | Change |
|--------|-------:|------:|-------:|
| `bp_us` | 2 457 µs | 769 µs | **−69 %** |

### Narrowphase: eliminate per-frame allocation

`par_iter().filter_map().collect::<Vec<_>>()` allocated a fresh
`Vec<ContactManifold>` every substep. Replaced with a pre-allocated
`produced_buffer` field on `NarrowphasePipeline` that is cleared and
extended via `par_extend` each substep, avoiding heap pressure under load.

| Metric | Before | After | Change |
|--------|-------:|------:|-------:|
| `np_us` | 519 µs | 121 µs | **−77 %** |

### Solver: sparse Jacobi clear/flush

The parallel Jacobi path was scanning all 10 000 body slots on every
clear and flush, even though only ~210 slots (2 per active manifold) are
touched. A `jacobi_touched_bodies` scratch buffer now tracks only those
indices: sorted, deduplicated, and iterated directly.

The `parallel_contact_push_threshold` was lowered from 256 to 16 contacts
so that typical 96-manifold scenes activate the parallel path.

| Metric | Before | After | Change |
|--------|-------:|------:|-------:|
| `sv_us` | 1 220 µs | 173 µs | **−86 %** |

### Sleep island manager: stride fix and early-skip

**Stride bug fix.** The sleep-update stride gate used the condition
`step_index % stride == 0 || step_index + 1 == substeps`. When
`substeps ≤ stride` both terms were simultaneously true, causing sleep to
run on every substep instead of only the last. Fixed to
`is_last || (substeps > stride && index % stride == 0)`.

**New body-count tier.** Added a 6 000-body threshold that increases the
update stride to 4, reducing sleep work for large scenes.

**Early-skip.** Bodies that are already awake with a zero sleep timer skip
the full update path with a single branch check.

| Metric | Before | After | Change |
|--------|-------:|------:|-------:|
| `sl_us` | 731 µs | 62 µs | **−92 %** |

### Combined physics step improvement

| Metric | Before | After | Change |
|--------|-------:|------:|-------:|
| `phys_us` (2 substeps) | ~6 000 µs | 2 174 µs | **−64 %** |

## MPS Async Physics Pipeline

### PhysicsMpsRunner

A new `PhysicsMpsRunner` type (`runtime/src/physics_mps_runner.rs`) wraps
`PhysicsWorld` in an `Arc<Mutex<>>` backed by a dedicated `MpsScheduler`.

```
PhysicsMpsRunner
  ├── world: Arc<Mutex<PhysicsWorld>>
  └── scheduler: Arc<MpsScheduler>        ← Critical / Performance workers
```

`step_begin(dt) -> PhysicsStepToken` submits `world.step(dt)` as a
`TaskPriority::Critical` / `CorePreference::Performance` MPS task and
returns immediately. `PhysicsStepToken::wait()` blocks on an `mpsc`
channel until the step completes.

### Pipelined frame loop

The render loop is restructured to overlap physics with GPU upload:

```
Frame N
  ① token[N-1].wait()          ← near-instant; physics was done during upload[N-1]
  ② reconcile_after_step       ← recycle escaped balls
  ③ build_frame_instances      ← read settled physics state  (par_iter)
  ④ draw_compiler.compile()    ← build draw batches
  ⑤ apply_runtime_patch
     + physics_tick            ← spawn/force setup for step N
     + step_begin(dt)  ────────────────────────────────┐
  ⑥ upload_draw_frame (~9 ms) ◄── GPU upload            │ physics[N] runs here
  ⑦ render                                             │
                                                       ↓
Frame N+1: token[N].wait() returns immediately  ───────┘
```

Physics (~3.6 ms, 5 substeps) completes well within the GPU upload window
(~9–38 ms depending on GPU load), so `token.wait()` at frame start adds
essentially zero overhead.

### Safety guarantees

- `reset_simulation_state()` drains any pending token before replacing the
  world, preventing use-after-reset.
- `prepare_for_exit()` drains any pending token before GPU teardown.

## Bounce Showcase — 30 000 Bodies

The default `target_ball_count` in `bounce_showcase.tlscript` is raised
from 8 000 to **30 000**. Scatter burst thresholds and gravity ramps are
tuned for the higher body count, demonstrating multi-core utilisation at
production scale.

At 30 000 bodies on a 24-thread desktop the physics substep cost scales
roughly linearly with body count while all cores remain active, confirming
the parallel pipeline is exercised at full width.
